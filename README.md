# 💰 콘솔 가계부 프로젝트 (Budget App)

> **"작은 서비스"**란 기능이 많은 게 아니라 예외 상황에서도 데이터가 안전한 것을 말합니다. 단순한 유틸리티를 넘어 유지보수 가능한 아키텍처와 엄격한 데이터 무결성을 목표로 설계된 파일 입출력 기반의 가계부 콘솔 프로그램입니다.

---

## 🏗️ 1. 아키텍처 및 모듈/클래스 책임 분리 (구조화)

본 프로젝트는 의존성이 한 방향으로만 흐르도록 설계된 계층형 구조를 따르며, 단일 책임 원칙(SRP)을 준수합니다.

### 1.1. 모듈 단위의 책임 분리
* **`models.py` (Domain)**: 가계부의 핵심 데이터 구조와 규칙을 정의합니다.
* **`storage.py` (Data Access)**: 파일 I/O 시스템과 직접 맞닿아 있으며, 데이터를 스트리밍으로 읽고 원자적으로 쓰는 역할만 수행합니다. 비즈니스 로직은 모릅니다.
* **`services.py` & `io_service.py` (Business Logic)**: CRUD 규칙 검증, 요약 통계 계산 등 핵심 비즈니스 로직을 담당합니다.
* **`__main__.py` (CLI Controller)**: 사용자의 입력을 파싱하고, 서비스를 호출하며, 그 결과를 화면에 이쁘게 출력하는 프레젠테이션 레이어입니다.

### 1.2. 최소 2개 이상의 클래스 책임 경계 (어떻게 정했는가?)
데이터를 담는 **상태(Model)** 클래스와 로직을 실행하는 **행동(Service)** 클래스로 책임을 엄격히 분리했습니다.
* **`Transaction` 클래스 (데이터/상태)**: 단순히 데이터를 담는 것을 넘어, `__post_init__` 매직 메서드를 통해 금액이 음수인지, 날짜 포맷이 올바른지 스스로 검증합니다. "데이터의 무결성 유지"가 유일한 책임입니다.
* **`TransactionService` 클래스 (행동/비즈니스)**: `Transaction` 객체의 생명주기를 관리합니다. 새 ID를 발급하고, 스토리지에 저장을 지시하며, 조건에 맞는 내역을 검색(`search`)하는 흐름을 제어합니다. 데이터 자체의 유효성 검증은 Model에게 위임합니다.

## 🛠️ 2. 핵심 기술 및 설계 의도 (Q&A)

### Q1. 파일 기반 update/delete를 "어떻게" 안전하게 처리했는가?
**A. `tempfile`과 `os.replace`를 활용한 원자적 덮어쓰기(Atomic Rewrite) 패턴을 적용했습니다.**
수정/삭제 시 `storage.py`의 `rewrite_records()` 함수가 호출됩니다. 
1. 원본 파일을 직접 수정하지 않고, `tempfile.mkstemp`로 임시 파일(`*.tmp`)을 생성합니다.
2. 삭제 대상은 제외하고(또는 수정 대상은 값을 변경하여) 임시 파일에 새롭게 데이터를 스트리밍하며 씁니다.
3. 쓰기가 완벽하게 끝난 후, OS 단의 `os.replace()`를 호출해 임시 파일을 원본 파일명으로 덮어씌웁니다. 
이로써 **작업 도중 정전이 되거나 강제 종료되어도 원본 파일이 깨지는 데이터 유실 현상을 원천 차단**했습니다.

### Q2. list/search를 제너레이터로 스트리밍 처리한 방식을 "어떻게" 구현했고, "왜" 유리한가?
**A. `yield` 키워드를 사용해 한 줄씩 데이터를 반환(`read_stream` 함수)하도록 구현했습니다.**
```python
# storage.py 구현부 발췌
with open(path, 'r') as f:
    for line in f:
        yield json.loads(line)
```
* **유리한 이유**: 파일 전체를 `list`로 메모리에 올리지 않고, 필요할 때마다 한 줄씩 디스크에서 읽어옵니다. 거래 내역이 10건이든 100만 건이든 **메모리(RAM) 사용량이 항상 O(1)로 일정하게 유지**되어 시스템 셧다운(MemoryError)을 방지할 수 있습니다.

### Q3. 데코레이터로 분리한 공통 기능은 무엇이며, "왜" 분리했는가?
**A. `@error_handler` (예외 처리 로직)와 `@time_logger` (실행 시간 측정)를 분리했습니다.**
* **어떻게**: `services.py` 상단에 래퍼(Wrapper) 함수를 만들어 핵심 로직(`func`) 실행 중 발생하는 `ValueError`, `KeyboardInterrupt` 등을 `try-except`로 낚아채어 친절한 한글 메시지로 변환하고 `sys.exit(1)`로 에러 코드를 반환하게 했습니다.
* **왜**: 비즈니스 로직 안에 `try-except` 코드가 섞이면 가독성이 심각하게 훼손됩니다. 데코레이터를 통해 **관심사(비즈니스 로직 vs 시스템 예외 처리)를 분리**하여, 개발자는 핵심 기능 구현에만 집중할 수 있게 되었습니다.

### Q4. 타입 힌트를 적용해 얻는 이점을 실제 코드 예로 설명한다면?
**A. 개발자의 실수를 컴파일 타임(IDE)에 방지하고, 코드 자체가 명세서 역할을 합니다.**
* **실제 예**: `def search(..., from_date: Optional[str] = None) -> Iterator[Dict[str, Any]]:`
* 위 코드를 통해 인자가 누락될 수 있음(`Optional`)을 명시하고, 리턴값이 단순 리스트가 아니라 `Iterator`임을 보장합니다. 따라서 이 함수를 호출하는 쪽에서 `.append()` 같은 리스트 전용 메서드를 쓰려고 하면 IDE(VS Code 등)가 즉각 빨간 줄로 경고를 띄워 런타임 에러를 사전에 막아줍니다.

### Q5. JSONL과 CSV 중 선택한 저장 포맷과 "왜" 그것을 택했는가?
**A. 핵심 스토리지 포맷으로 `JSONL (JSON Lines)`을 선택했습니다.**
* **이유**: 가계부 데이터에는 단순 문자/숫자뿐 아니라 **태그(`tags`)와 같은 배열(List) 데이터**가 포함됩니다. CSV는 배열을 담으려면 쉼표로 묶어(`"a,b"`) 문자열로 저장한 뒤 다시 분리(`split`)해야 하는 파싱 오버헤드와 타입 캐스팅(금액 문자를 int로 변환) 문제가 발생합니다. 반면 JSONL은 파이썬의 `list`, `int`, `str` 타입을 네이티브하게 그대로 직렬화/역직렬화할 수 있어 데이터 구조의 변형 없이 안전하게 저장할 수 있기 때문입니다.

### Q6. 거래가 10만 건으로 늘어난다면 병목은 어디이며, 어떻게 개선할 것인가?
**A. 데이터 변경(`update/delete`) 시 발생하는 전체 파일 재기록(Full Rewrite)이 병목이 됩니다.**
* 현재는 원자성을 위해 1건을 수정해도 10만 건 전체를 새로 써야 하는 O(N) 비용이 발생합니다.
* **개선 방안**: 파일 단위 저장을 유지해야 한다면, 월별/연도별로 파일을 분할(Sharding) 저장하여(예: `transactions_2024_01.jsonl`) 한 번에 읽고 쓰는 파일의 크기를 줄이겠습니다. 더 고도화한다면 파일 I/O를 대체하여 B-Tree 인덱싱을 지원하는 `SQLite`와 같은 로컬 DB로 전환하는 것이 근본적인 해결책입니다.

### Q7. import CSV에 일부 깨진 행이 섞이면 어떻게 처리하여 사용자 신뢰를 지킬 것인가?
**A. All-or-Nothing 대신 "부분 성공(Partial Success) 및 리포트" 전략을 취했습니다.**
* **어떻게**: `io_service.py`의 `import_csv` 내부를 보면 전체 루프가 아닌 **각 행(Row) 단위로 `try-except` 블록**을 씌워두었습니다.
* 파일 내 100줄 중 1줄의 금액에 문자가 섞여 에러가 나더라도, 프로그램이 뻗지 않고 정상적인 99줄은 `imported_count`로 반영하고, 실패한 1줄은 `skipped_count`로 집계하여 최종 결과로 `[완료] imported=99, skipped=1` 형태로 투명하게 리포트합니다.
